# 가족 약 복용 안전 관리 서비스 기획안(상세본)

> **한 줄 정의**: 가족 구성원이 질병/진단 정보는 공유하지 않고, 복용 중인 약의 **성분·주의사항만 선택적으로 공유**하여 일상에서 발생하는 **중복 복용·상호작용 가능성**을 빠르게 확인할 수 있는 약 안전 정보 서비스

## 1. 기획 배경 및 시장/사회 맥락

- 고령화·만성질환 증가로 다약제(여러 약 동시 복용) 가구가 증가
- 자녀/보호자가 대신 약을 구매하거나 복용 여부를 챙기는 상황이 빈번
- 일반 사용자는 약의 성분 중복, 상호작용, 주의군(임산부·고령자·간/신장) 등을 스스로 판단하기 어려움
- 기존 약 정보 서비스는 개인 관리 또는 전문가 중심이 많아, 가족 돌봄 관점의 선택적 공유 UX가 상대적으로 약함

---

## 2. 문제 정의(사용자 관점)

- 가족이 복용 중인 약 성분을 모른다 → 같은 성분의 약을 추가 구매/복용할 위험
- 상호작용 가능성을 즉시 확인하기 어렵다(특히 약국/편의점/집)
- 가족이라도 병명·진단을 공유하는 것은 부담(개인정보/낙인/관계 갈등)
- “안전 여부”가 아니라도 최소한 주의가 필요한지 알려주는 도구가 부족

---

## 3. 목표 및 원칙

### 서비스 목표

- 가족의 중복 복용/상호작용 위험 ‘가능성’을 낮춘다
- 돌봄 상황에서 빠르게 확인할 수 있게 한다(구매 직전/복용 직전)
- 개인정보 노출 최소화(병명·진단·처방 사유 입력/공유 배제)

### 설계 원칙(Non-negotiables)

- **정보 제공형(Info-only)**: 의료 판단/처방 권고가 아니라 “가능성·주의 정보 + 다음 행동(약사/의사 문의)” 제공
- **선택적 공유(Opt-in)**: 기본 비공개, 사용자 동의 기반 공유/철회
- **최소 데이터**: 실명·병명·진단정보 수집하지 않음(닉네임 중심)
- **명확한 근거 표기**: “어떤 성분/어떤 조합 때문에 주의가 필요한지”를 짧게라도 보여줌

---

## 4. 타겟 사용자 및 페르소나

### 1차 타겟

- 부모/조부모를 돌보며 대신 약을 사는 자녀/보호자
- 만성질환으로 여러 약을 복용하며 복용 안전성을 점검하고 싶은 사용자

### 대표 페르소나(예시)

- 보호자형(자녀, 32세): “엄마가 무슨 약 먹는지 정확히 몰라서 같은 성분을 또 살까 봐 불안”
- 다약제 사용자(60대): “약이 많아서 내 약 목록을 정리해두고, 필요한 사람에게만 보여주고 싶다”
- 동거 가족(배우자): “응급 상황에서 최소한의 정보(약 성분)만 빠르게 확인하고 싶다”

---

## 5. 핵심 가치 제안(USP)

- 가족 공유에 최적화된 약 정보 서비스
  - 건강상태(병명)가 아니라 약 성분만 공유
  - 대신 약 사주기 상황에 특화된 짧은 플로우
  - 결과는 “판정”이 아닌 가능성 + 근거 + 행동 가이드로 제공

---

## 6. 서비스 범위(Scope)

### 포함(초기/MVP)

- 약 등록(검색/스캔/직접 입력 보조)
- 성분/주의사항 자동 정리(공개 API 기반)
- 가족 그룹 생성 및 초대(링크/코드)
- 약 정보 선택 공유(가족별 권한)
- 새 약 입력 시 중복 성분/상호작용 가능성 안내(정보 제공형)
- “대신 약 사주기” 모드(빠른 확인)

### 제외(초기 단계에서 의도적으로 보류)

- 복용 알림/복약 스케줄 자동화(확장 기능으로)
- 질병/진단/검사 수치 기반 맞춤 권고(의료영역 확장 리스크 큼)
- 처방전 자동 판독(기술/규제/품질 이슈)

---

## 7. 핵심 사용자 시나리오(유저 플로우)

### 시나리오 A: “대신 약 사주기” (핵심)

1. 보호자가 앱에서 가족의 공유된 약 목록 확인
2. 약국에서 구매하려는 약을 이름 검색/바코드 스캔
3. 앱이 가족의 현재 약과 비교하여 아래를 요약 표시
   - 성분 중복 가능성
   - 상호작용 가능성
   - 주의가 필요한 군(가능한 범위 내)
4. 결과와 함께 “약사에게 이렇게 물어보세요” 문장 제공(행동 가이드)
5. 필요 시 해당 가족에게 “이 약 사도 되는지?” 메시지/확인 요청(선택 기능)

### 시나리오 B: “내 약 등록 후, 필요한 가족에게만 공유”

1. 사용자가 약 이름으로 등록
2. 성분/주의사항 확인
3. 공유할 가족 선택 → ‘성분/주의사항만 공유’ 설정
4. 언제든 공유 철회/범위 조정

---

## 8. 정보 구조(IA) 및 화면 구성(초안)

- 홈
  - 내 약 요약(등록 개수, 최근 등록)
  - 가족 그룹 바로가기
  - 대신 약 사주기 모드 진입
- 내 약
  - 약 목록(검색/필터)
  - 약 상세(성분/주의 요약, 출처, 업데이트 일시)
  - 공유 설정(이 약을 누구에게 공유?)
- 가족
  - 가족 그룹/구성원
  - 가족별 “공유된 약” 보기(허용된 범위만)
  - 권한/초대/철회
- 대신 약 사주기
  - 약 입력(검색/스캔)
  - 비교 결과(중복/상호작용 가능성 카드)
  - “약사에게 질문하기” 텍스트/체크리스트
- 설정/안내
  - 개인정보/약관/고지
  - 데이터 출처/정확도 안내
  - 문의/신고(데이터 오류, 약 정보 수정 요청)

## 9. 기능 상세 기획(PRD 수준)

### 9-1. 약 등록

- 입력 방식
  - 약 이름 검색(자동완성)
  - 바코드/제품코드 스캔(가능 시)
  - 직접 입력(예외 케이스: 해외약/건기식 → “비공식” 표시)
- 저장 항목(최소)
  - 약명(표준명/표시명)
  - 주요 성분(표준 성분명, 함량 가능 시)
  - 주의사항 요약(금기/주의/병용주의 등)
  - 출처(공개 API/사용자 입력)
  - 등록일/수정일
- UX 포인트
  - 등록 시 “병명/복용 이유 입력 없음”을 명확히 안내(프라이버시 신뢰)

### 9-2. 가족 초대 및 공유 권한

- 그룹 모델
  - 그룹 생성(예: “우리 가족”)
  - 초대 링크/코드(유효기간, 재발급)
- 공유 권한 단위
  - 기본: 약 단위 공유(이 약을 A에게 공유)
  - 옵션: “내 약 전체 공유”도 가능하되, 기본은 OFF
- 조회 제어
  - 공유되지 않은 약은 존재 자체가 보이지 않도록(메타정보 최소화)

### 9-3. 가족 약 목록 확인

- 표시 정보(공유 허용된 범위만)
  - 약 이름
  - 주요 성분
  - 주의사항 요약(짧게)
  - 데이터 출처/갱신일(작게)
- 제외 정보
  - 병명, 처방 사유, 복용 시간/빈도(초기 범위 제외)

### 9-4. 중복·상호작용 “가능성” 안내(Info-only 정책)

- 입력: 새로 구매/복용하려는 약(또는 성분)
- 비교 대상: 내 약 + (권한 있는) 가족 약
- 출력 형태(권장 카드 구성)
  - “중복 성분 가능성: 있음/없음/확인불가”
  - “병용 주의 가능성: 있음/없음/확인불가”
  - 근거: 겹치는 성분명/주의 키워드
  - 행동 가이드: “약사에게 현재 복용 중인 ○○ 성분/약을 보여주고 병용 가능 여부를 확인하세요”
- 중요 UX
  - 항상 “확정 판정이 아니다”를 짧게 포함
  - “확인불가” 상태(데이터 누락/비표준 약)는 명확히 표시

### 9-5. 대신 약 사주기 모드(Quick Flow)

- 목표: 10~20초 내 결론성 정보 확인
- 단계 최소화: 약 입력 → 비교 결과 → 공유 약 목록/근거 확인(접기/펼치기)
- 오프라인/저신호 고려(가능하면 캐시)
- 매장 상황 고려: 글자 크게, 대비 높게, 버튼 큼

## 10. 데이터/기술 개요(기획 관점)

### 데이터 소스

- **공개 API(식약처/공공데이터 등) 기반**을 1차로 사용
- (웹 리서치 반영) 국내에서는 **의약품안전나라(식약처)**의 의약품 정보(성분/주의사항 등)와, **DUR(의약품안전사용서비스)** 기반의 병용금기/주의 체계를 주요 레퍼런스로 삼는다.

### 데이터 품질 이슈 대응

- 표준화: 약명/성분명 정규화(동의어, 표기 변형)
- 갱신 주기: 일/주 단위 배치 업데이트 + 사용자 신고 반영 프로세스
- 커버리지: 일반의약품/전문의약품/건기식/해외약을 구분 표시

### 데이터 신뢰 표시(권장)

- “공개 데이터 기반(갱신일: YYYY-MM-DD)”
- “사용자 입력(검증 전)” 배지
- (웹 리서치 반영) “DUR 기준 참고(출처/라이선스 범위 내)”
- (웹 리서치 반영) “정부 서비스 ‘내가 먹는 약! 한눈에’(개인 조회)와는 별개로, 본 서비스는 가족 간 선택 공유 UX를 제공”

## 11. 개인정보 보호 및 신뢰 설계(상세)

- 수집 최소화
  - 실명 대신 닉네임/프로필 이미지 선택
  - 병명/진단/복용 목적 입력 항목 배제
- 권한/감사
  - 누가 내 약 정보를 조회했는지 조회 이력 제공(신뢰 강화)
  - 공유 철회 시 즉시 접근 차단
- 보안(요구사항)
  - 전송/저장 암호화
  - 초대 링크 토큰 만료/재발급
- 고지
  - “의료 조언이 아닌 정보 제공” 고지
  - 이상반응/응급 시 행동(119/의료기관) 안내 문구
  - (웹 리서치 반영) 상호작용/금기 정보는 국가 DUR 체계 등 공신력 있는 기준을 참고하되, **개인 상태에 따라 달라질 수 있어 전문가 상담이 필요**함을 기본 원칙으로 고지

## 12. 운영 정책(필수)

- 데이터 오류 신고: 약 상세 화면에서 “정보 오류 신고”
- 콘텐츠 관리: 주의사항 요약 문구는 과장 금지, 출처 명시
- 분쟁/오남용 대응: 가족 관계 악용(무단 초대/압박) 방지 가이드 및 차단 기능(추후)
  - (웹 리서치 반영) 공공 데이터/외부 DB를 사용하는 경우, **출처 표기/업데이트 주기/라이선스 준수**를 운영 체크리스트로 고정

---

## 17. 국내 제도·레퍼런스(웹 리서치)

- 국내에는 개인의 투약 이력을 조회하는 공공 서비스인 **‘내가 먹는 약! 한눈에’(gov.kr 안내)**가 존재한다.
  - 본인 인증 기반으로 개인의 복약(조제) 이력을 확인하는 성격이 강함
  - 본 기획은 개인의 “투약 이력” 자체를 공유하지 않고, **가족에게 필요한 최소 정보(약 성분/주의사항)만 선택적으로 공유**하는 UX로 차별화
- **DUR(의약품안전사용서비스)**는 병용금기/연령·임부 등 주의 정보를 의료현장에서 점검하기 위한 국가 수준의 체계로, 본 서비스의 “가능성 안내” 근거 체계를 설계할 때 레퍼런스로 활용한다.

---

## 18. 데이터 신뢰성 및 출처 정책(웹 리서치 반영)

- **출처 우선순위(국내)**:
  - 1순위: 식약처 **의약품안전나라**(성분/주의사항 등)
  - 2순위: 공공데이터포털/공공기관 제공 데이터(예: 병용금기 데이터 파일 등)
  - 3순위: 사용자 입력(해외약/건기식 등) → “검증 전/확인불가”로 명확히 라벨링
- **표시 원칙**:
  - 약 상세 화면에 “출처/갱신일/확인 가능 범위”를 항상 표시
  - 데이터 미포함(조회 불가)인 경우 “확인불가” 상태로 강등하고, 다음 행동(약사 문의)을 함께 제시
- **글로벌 확장 원칙**:
  - 국가별로 의약품 코드/성분 표준이 달라 동일 UX를 유지하기 어렵기 때문에,
  - 초기에는 “한국 공식 출처 기반”을 명시하고, 해외는 “사용자 입력 + 확인불가 처리”를 기본값으로 설계한다.

---

## 19. DUR 기반 상호작용(가능성) 안내 정책(웹 리서치 반영)

> 목표: “판정”이 아니라 “**가능성 + 근거 + 다음 행동**”을 일관되게 제공한다.

- **결과 상태(권장 3단계)**:
  - **주의 필요**: 병용금기/병용주의 등으로 알려진 조합(또는 중복 성분)이 존재할 가능성이 높음 → “약사/의사 상담” CTA 고정
  - **현재 정보로는 큰 이슈 없음**: 확인 가능한 범위에서 중복/금기 근거를 찾지 못함 → “단, 개인 상태에 따라 달라질 수 있음” 고지
  - **확인불가**: 데이터 미포함/표준화 실패/사용자 입력 약 등 → “제품명/성분표를 약사에게 제시” 가이드
- **근거 제시 템플릿(예시)**:
  - “겹치는 성분: ○○”
  - “주의 키워드: 병용금기/병용주의/연령/임부(가능한 범위 내)”
- **예외 안내 원칙**:
  - DUR 체계에서도 전문의 판단에 따라 예외 처방이 가능할 수 있음을 전제로, 서비스는 “금지”가 아니라 “상담 유도”를 기본으로 한다.

## 20. 공공데이터/라이선스 리스크 및 대응

- (웹 리서치 반영) 일부 공공 데이터(예: 병용금기 데이터)는 **비상업적 이용 범위**가 명시되거나, 상업적 이용 시 별도 승인/협약이 요구될 수 있다.
- **대응 원칙**:
  - MVP 단계: 데이터 이용조건을 문서화하고, 서비스 화면/약관에 출처·이용범위를 표기
  - 상용화 단계: 데이터 제공기관과 **정식 승인/계약/라이선스** 확보를 전제로 기능 범위를 확정
  - 라이선스 불확실 시: 해당 기능은 “확인불가” 처리 + 약사 문의 UX로 우회

## 14. MVP 로드맵

### Phase 1 (MVP)

- 약 등록(검색 중심) + 성분/주의 요약
- 가족 초대 + 약 단위 공유
- 대신 약 사주기 모드 + 중복/상호작용 “가능성” 안내(Info-only)

### Phase 2

- 스캔/바코드 정확도 개선
- 복용 이력(선택), 개인 약 히스토리
- 고령자 UX(큰 글씨 모드, 보호자 대리 설정)

### Phase 3

- 복약 알림/복용 체크(돌봄 기능 강화)
- 약 콘텐츠/캠페인(주의사항 교육)
- 추가 데이터 소스 확장(커버리지/정확도 고도화)
